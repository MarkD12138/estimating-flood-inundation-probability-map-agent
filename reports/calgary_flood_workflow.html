Calgary Flood Inundation Probability Modeling Workflow

1 Setup / libraries / reproducibility notes

This report builds a Calgary-only flood inundation probability workflow in R. It creates a 60 m fishnet, derives the binary inundation target, engineers all required features, performs exploratory checks and minor cleaning, fits a glm logistic regression model, and saves validation outputs for later reuse.

# Source each helper script into its own environment so internal utility names do not collide.
load_stage_function <- function(script_name, fun_name) {
  stage_env <- new.env(parent = globalenv())
  sys.source(file.path(project_root, "R", script_name), envir = stage_env)
  stage_fun <- stage_env[[fun_name]]
  if (!is.function(stage_fun)) {
    stop(sprintf("Expected function `%s` in script `%s`.", fun_name, script_name))
  }
  environment(stage_fun) <- stage_env
  stage_fun
}

The inundation raster behaves as a binary target layer with 0 and 1 as valid values and a separate excluded code. The surface-water raster is also binary, so the hydrology step can compute water_cover_area without further class-mapping assumptions.

11 Key findings / discussion

The Calgary workflow is fully modular: every stage writes an intermediate artifact keyed by id, and the final modeling table preserves the same predictor names that will be needed later for City X transfer. The workflow also makes all major assumptions explicit, including the conservative land-cover mapping, the binary raster handling rules, and the separate thresholds used for target creation versus classification.

The strongest practical value of this report is transparency. Fishnet creation, target generation, feature engineering, EDA, cleaning, modeling, and diagnostics are all separated into named helper stages, which makes reruns and future debugging much easier than a single monolithic script.

12 Limitations and next steps

This Calgary-only workflow still has two important limitations. First, the watershed stage is computationally heavy because it relies on centroid-based hydrologic routing at city scale. Second, the final model quality should be interpreted with the usual caution for transfer learning: even a well-performing Calgary model may require recalibration before it is applied in another city.

The next step for City X deployment is to reproduce the same predictor set with the same variable names and the same preprocessing contract. Once those City X features exist, the saved scaling rules and glm object can be used to produce probability surfaces without changing the Calgary training workflow.
